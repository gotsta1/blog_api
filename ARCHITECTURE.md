# Архитектура и подход к кешированию

## Схема архитектуры

```
┌──────────┐      ┌──────────────┐      ┌───────────┐
│  Client   │─────▶│   FastAPI    │─────▶│  Redis    │
│ (HTTP)    │◀─────│   Server     │◀─────│  Cache    │
└──────────┘      │              │      └───────────┘
                  │              │
                  │              │      ┌─────────────┐
                  │              │─────▶│ PostgreSQL  │
                  │              │◀─────│   (DB)      │
                  └──────────────┘      └─────────────┘
```

## Стратегия кеширования: Cache-Aside

### Почему Cache-Aside?

Было требование: При GET /posts/{id} проверять наличие поста в Redis, если нет — брать из PostgreSQL и класть в кеш

1. **Простота** — логика явная и прозрачная, легко отлаживать
2. **Отказоустойчивость** — если Redis недоступен, приложение
   работает напрямую с PostgreSQL
3. **Эффективность** — в кеш попадают только запрашиваемые посты
4. **Консистентность** — явная инвалидация при записи/удалении

### Как это работает

**Чтение (GET /posts/{id}):**
```
1. Проверить Redis по ключу post:{id}
2. Если найден (cache hit):
   - Вернуть данные из кеша (0 SQL-запросов)
3. Если не найден (cache miss):
   - SELECT из PostgreSQL (1 SQL-запрос)
   - Сохранить пост в Redis с TTL 300 сек
   - Вернуть клиенту
```

**Запись (PUT /posts/{id}):**
```
1. Обновить запись в PostgreSQL
2. Удалить ключ post:{id} из Redis (инвалидация кеша)
```

**Удаление (DELETE /posts/{id}):**
```
1. Удалить запись из PostgreSQL
2. Удалить ключ post:{id} из Redis
```

### Redis-ключи

| Ключ | Назначение |
|------|------------|
| `post:{id}` | Кешированные данные поста (JSON) |

### Отказоустойчивость (Graceful Degradation)

Все вызовы к Redis обёрнуты в `try/except RedisError`.
Если Redis недоступен:
- `get_cached_post` → `None` (cache miss), данные из PostgreSQL
- `set_cached_post` / `invalidate_cached_post` → пропускается

Приложение продолжает работать без Redis, просто без кеша.

### TTL как страховка

TTL в 300 секунд гарантирует, что устаревшие данные не будут
жить в кеше бесконечно, даже если инвалидация не сработает.

### Почему не кешируется список постов

Список постов (`GET /posts`) не кешируется, потому что:
- Инвалидация при создании/обновлении/удалении — сложная задача
- Пагинация создаёт множество вариаций ключей
- Для MVP это избыточная оптимизация

### Альтернативы, которые были рассмотрены

| Паттерн | Плюсы | Минусы | Почему не выбран |
|---------|-------|--------|-----------------|
| **Write-Through** | Кеш всегда актуален | Замедляет запись | Избыточен для блога |
| **Write-Behind** | Быстрая запись | Риск потери данных | Слишком сложно |
| **Read-Through** | Похож на Cache-Aside | Требует провайдер | Нет преимуществ |

## Структура БД

```sql
CREATE TABLE posts (
    id SERIAL PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX ix_posts_created_at ON posts(created_at);
```

## Структура проекта

```
app/
├── main.py          — точка входа, lifespan-события
├── config.py        — настройки из .env (pydantic-settings)
├── database.py      — SQLAlchemy async engine и сессия
├── redis.py         — Redis-клиент
├── models.py        — SQLAlchemy-модель Post
├── schemas.py       — Pydantic-схемы для валидации
├── cache.py         — хелперы для работы с кешем
└── routers/
    └── posts.py     — CRUD-эндпоинты
```
